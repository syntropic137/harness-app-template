# EXP-05 backend-feedback

Claim: CobaltCoast (claude-opus-4-7), 2026-06-10, branch feat/apss-integration.
Sole-editor; do not edit this file from other agents.

## Mission slice
Hit the API directly, read structured responses; verify there is a DOCUMENTED
path from "I am a fresh agent" to "I know what endpoints exist and how to
call them".

## Frozen hypothesis
1. The bare template ships NO application-level HTTP API. All three
   `ws_apps/example-*` are one-shot emit-and-exit telemetry samples; their
   only "endpoints" are their telemetry exports.
2. The only API surface a fresh agent CAN hit directly today is the
   observability stack: VictoriaTraces (Jaeger-shaped), VictoriaLogs (LogsQL),
   and the OTEL Collector OTLP receivers.
3. That observability API surface IS documented for agents via the
   `observability-queries` skill in `.claude/skills/`, NOT via AGENTS.md
   itself. AGENTS.md lists the skill by name but the actual endpoint
   surface lives only inside the skill body.
4. There is NO Swagger / OpenAPI / introspection endpoint for the
   observability stack itself; agents discover the API shape only by
   reading the skill body or the upstream VictoriaMetrics docs.

## Frozen prediction
- 1 PASS: zero HTTP server frameworks in any example app's deps.
- 2 PASS: observability stack ports respond.
- 3 PASS: AGENTS.md gives only a skill pointer; skill body has the curl
  examples.
- 4 PASS: no OpenAPI introspection ships in the template.

Combined verdict prediction: CONFIRMED. The "backend API" axis is gated on
the agent INVOKING the observability-queries skill, not on AGENTS.md alone.

## Probes
- `grep -rEi "express|fastify|hono|koa|axum|actix|fastapi|flask|gin|net/http" ws_apps/example-*/` to confirm no HTTP server framework.
- Curl each documented observability endpoint with a malformed request and
  capture the error shape: is it structured JSON, or freeform text?
- Find the `observability-queries` skill body and grep for the actual
  documented endpoints.
- `find . -path ./node_modules -prune -o -name "openapi*.yaml" -o -name "openapi*.json" -print` to test 4.

## Observed (probes run 2026-06-10 01:50 UTC)

### P1 no app-level HTTP API
- `grep -rEi "express|fastify|hono|koa|axum|actix|fastapi|flask|gin|net/http|warp|rocket" ws_apps/example-*/` returned ZERO real matches (only false-positive substring hits in protected-config comments and the word "honors").
- All three example apps are one-shot emit-and-exit, confirmed in their package.json/pyproject.toml/Cargo.toml. No HTTP server framework imported anywhere.

### P2 observability surface responds
- VictoriaTraces Jaeger API: `GET /select/jaeger/api/services` -> `{"data":["example-typescript"], ...}` (structured JSON).
- VictoriaMetrics: `GET /api/v1/query?query=up` -> `{"status":"success","data":{"resultType":"vector","result":[]}, ...}` (Prometheus-shaped JSON).
- OTEL Collector OTLP HTTP: `POST /v1/traces` with `{}` -> `{"partialSuccess":{}}` (structured JSON).
- VictoriaLogs LogsQL malformed query: empty body (no structured error). Matches the EXP-03 log path opacity.

### P3 AGENTS.md vs skill body
- AGENTS.md line 31 references the `observability-queries` skill by name but contains ZERO actual endpoint URIs.
- `.claude/skills/observability-queries/SKILL.md` is the real surface; lines 23-25 declare the VL_PORT/VM_PORT/VT_PORT env-var convention and the curl examples for `/select/logsql/query`, `/api/v1/query`, `/select/jaeger/api/traces`, `/select/jaeger/api/services` start at lines 52-150.
- A fresh agent that does NOT invoke the skill cannot get to the endpoints from AGENTS.md alone.

### P4 no OpenAPI / Swagger surface
- `find . -name openapi.* -o -name swagger.*` returned zero results.
- OTEL Collector HTTP root returns plain 404 with `Content-Type: text/plain`, no introspection.
