## Mission slice
Change a backend response, observe it; change a UI element, observe it. Score
how discoverable + how fast the inner loop is in this harness.

Operating definitions for this template:
- "backend" = `ws_apps/example-typescript` (emits one span + one Pino log on
  each `pnpm start`). The "response" surface is the structured log line and
  the span body that lands in VictoriaLogs / VictoriaTraces.
- "UI" = `ws_apps/docs` (Next.js + Fumadocs, `next dev -p 3001`). The
  "element" is a Markdown / MDX content node.

## Frozen hypothesis
1. Backend feedback loop is OBSERVABLE without docker only if the telemetry
   stack is up. `pnpm --filter @example/typescript start` will exit cleanly
   after one emit (it is a one-shot, not a server). The "response" is
   visible only via the VictoriaLogs query, NOT via stdout shape alone.
2. UI feedback loop via `pnpm --filter @harness/docs dev` will be reachable
   on http://localhost:3001 in under 15 seconds of dev-server startup, with
   HMR on MDX edits.
3. Neither feedback loop is documented from AGENTS.md alone. The agent has
   to read both `ws_apps/example-typescript/README.md` AND `ws_apps/docs/`
   contents to know what the "inner loop" even is. Docs/ has NO README.
4. The harness fast-path for "change UI, see UI" is `next dev -p 3001` and
   takes seconds; the fast-path for "change backend, see telemetry" is
   gated by docker-compose stack boot (tens of seconds first time).

## Frozen prediction
- 1 PARTIAL: backend emit succeeds, observability roundtrip requires stack.
- 2 PASS: next dev boots, hits 3001 within 15s.
- 3 PASS: zero AGENTS.md mention of `ws_apps/docs` or a UI dev loop.
- 4 PASS: dev loop is fast; telemetry loop is slow on first boot.

Combined verdict prediction: PARTIAL. Fast feedback EXISTS but is undocumented
at the AGENTS.md layer; the gates are infra (docker) and docs gaps.
