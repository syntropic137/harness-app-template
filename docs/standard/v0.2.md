## 1. Mental model

The harness is a **tool belt with named slots**. Each slot is a labeled hole; a **plugin** is something that fills a slot by satisfying its contract. A **template** is a specific set of plugins (one per slot) plus workspace conventions, packaged so a new project can be scaffolded in one command.

```
                          ┌──────────────────────────┐
                          │   Template (polyglot-    │
                          │   monorepo, rust-ws, …)  │
                          └──────────┬───────────────┘
                                     │  composes
                                     ▼
   ┌────────────────────────────────────────────────────────┐
   │                       Tool Belt                        │
   │                                                        │
   │  [stack-manager] [inspector]  [hooks]  [telemetry-sdk] │
   │  [observability-stack]  [sensors]  [agent-plugins]     │
   │  [task-runner]  [secret-scanner]  [doc-validator]      │
   │  [versioning]                                          │
   └────────────────────────────────────────────────────────┘
                                     ▲
                                     │  fills
                          ┌──────────┴────────────────┐
                          │   Plugins (lefthook,      │
                          │   pino, otel-node, …)     │
                          └───────────────────────────┘
```

Plugins are independently swappable. A polyglot template can mount multiple `telemetry-sdk` plugins (one per language) without changing the Standard.

## 2. Cross-cutting principles

These apply to every slot. They are non-negotiable in v0.2.

| Principle | Rule |
|---|---|
| **Polyglot-first** | Slot *contracts* MUST NOT assume any specific language for the consumer of a plugin. A TS app and a Python app must be able to use the same shipped plugin equally well. (Plugin *implementations* MAY be opinionated — see "Rust-first" below.) |
| **Rust-first for harness tools** | Where a slot is a tool (not an SDK), the reference plugin SHOULD be a single Rust binary. Rationale: trivially cross-platform, no runtime dependency, performant, single artifact to ship. Exceptions: `telemetry-sdk` is intrinsically per-language; `inspector` may need to wrap Playwright (research pending). Recorded per slot in §4. |
| **Cross-platform** | macOS + Linux are first-class. Windows is supported when reasonable. Plugins MUST NOT ship `.sh` or `Makefile` as primary entrypoints. `justfile` is the canonical task runner. Anything beyond simple shell needs a script in a language the project already uses (or Rust). |
| **Pragmatic vendor primacy** | `.claude/` is the canonical agent-plugin source dir and `CLAUDE.md` is the canonical agent-context file. Other vendors' conventional names (`AGENTS.md`, `.codex/`, `.cursor/`, `GEMINI.md`, etc.) are symlinks pointing AT the canonical files. We optimize for the primary user; other vendors interop via the symlinks. |
| **Plug-and-play** | Optional capabilities (e.g., the `sensors` slot today) must be cleanly removable. Templates SHOULD support `just <slot> init` recipes to add optional plugins after scaffold, and removal must not break the rest of the harness. |
| **Evidence-driven** | A plugin earns its slot via a go-verdict experiment. A template earns its name via a separate experiment proving the composition works end-to-end. Promotion criterion is documented per slot. |
| **Token-aware** | Plugins SHOULD provide token-efficient default outputs for agents. Verbose modes are opt-in. |
| **Testing pyramid + protected config** | Every plugin enforces a three-layer testing contract (unit + integration + end-to-end) with thresholds that are mechanically enforced (vitest `thresholds`, `cargo llvm-cov --fail-under-*`, `pytest --cov-fail-under`, etc.). Threshold values carry a "HARNESS-ENGINEERING PROTECTED CONFIG / DO NOT ADJUST" sentinel comment block. If a threshold needs adjustment, file a hypothesis-first experiment per CLAUDE.md, do not silently relax. See §2.5 below for the full pyramid spec. |
| **Eat our own dogfood** | The harness MUST instrument and govern itself. `harness-sensors gate harness/` runs against the harness's own code with a `.harness/governance.toml` policy. The harness's own quality score is a load-bearing signal — if our governance degrades, the gate trips on us first. **This template's eat-own-dogfood seed is committed at [`../../harness/.harness/governance.toml`](../../harness/.harness/governance.toml)** (ported verbatim from the lab; bead `create-harness-app-n48.4`). |

### 2.5. Testing pyramid (mechanically enforced)

Three layers, each with a distinct contract and a distinct enforcement mechanism. Drawn from [`software-leverage-points:testing`](https://github.com/syntropic137/software-leverage-points) — invoke that skill when designing a new plugin's test suite.

| Layer | What it covers | Enforcement |
|---|---|---|
| **Unit (100% target)** | Pure functions, single-module behavior, branching logic. Runs without I/O, without containers, without external deps. | Vitest / cargo test / pytest with coverage thresholds in config: lines/functions/statements 100%, branches ≥90%. PROTECTED config, see §2 above. |
| **Integration (100% surface)** | Public CLI / API surface of a plugin. Tests every documented subcommand and every documented config-key path. Spawns the actual binary; may use temp dirs and mocked external services. | `tests/integration/**` directory per plugin; same coverage tool; same threshold sentinel. |
| **End-to-end (smoke, gated)** | The plugin running against a real composed environment (live observability stack, real Docker daemon, real network). Catches the bug classes invisible to unit + integration tests — see retrospective 021 (lab) for the canonical example (BatchSpanProcessor flushed-too-late). | Lives under `experiments/<date>--<slug>--smoke/`. Run manually or via `just <slot> smoke`. Treated as a *probe*, not a CI gate — failures inform the next probe, they don't block merges. |

**Why all three.** Unit coverage to 100% guarantees that **lines are reached**, not that **functionality works**. The line `await fetch(url)` can be 100% covered by a unit test that mocks `fetch` — the call never crosses a network boundary, and a real network failure (DNS, TLS, timeout, malformed-response) goes undetected. Integration tests cover the surface a downstream consumer sees but still run in a single process. Only an end-to-end probe catches the "actually crossing the wire" bug class.

**Plugins MAY skip a layer if it's structurally inapplicable** (e.g., the `task-runner` slot's plugin is `just` itself; we don't author it, so there's no unit suite to write). When skipping, the plugin's slot doc states which layers don't apply and why.

## 3. Workspace conventions

Required directory layout at template root:

```
ws_apps/             # deployable / runnable units; each subdir picks its own stack
ws_packages/         # shared libs / cross-cutting code; same rule
harness/             # slot plugins (Rust-first where possible)
infra/               # observability-stack plugin (compose, configs) — note: this fork
                     #   inlines it under harness/observability/ instead, per ADR-0005
docs/                # per-template docs + ADRs + retrospectives
.claude/             # CANONICAL agent-plugin dir (skills/commands/agents/hooks)
CLAUDE.md            # CANONICAL agent context (root)
justfile             # task-runner plugin
```

Vendor symlinks (created by the template's `just agents link` recipe; Windows fallback deferred to v0.3):

```
AGENTS.md  -> CLAUDE.md     # CLAUDE.md is canonical
CODEX.md   -> CLAUDE.md
GEMINI.md  -> CLAUDE.md
.codex     -> .claude       # .claude/ is canonical
.cursor    -> .claude
```

The `ws_` prefix groups workspace dirs visually and alphabetically. Non-workspace top-level dirs do not get the prefix.

## 4. Slot contracts (v0.2)

Each slot defines:
- **Purpose** — what the plugin is for
- **Required interface** — what it must expose
- **Discovery** — how the harness finds and invokes it
- **Promotion criterion** — what evidence promotes a candidate to "shipping plugin"
- **Local pick** — the ADR in this template that records the specific plugin

This template's concrete manifest binding lives in [`../../harness.manifest.json`](../../harness.manifest.json), with a shorter contract summary and swap example in [`../slot-contracts.md`](../slot-contracts.md).

### 4.1 `stack-manager`

- **Purpose:** allocate ports, generate compose files, boot/stop/destroy/inspect the dev stack.
- **Required interface:** a CLI exposing `boot`, `stop`, `destroy`, `inspect` subcommands. `inspect` MUST emit a JSON object on stdout containing at least `{ project, ports: { ... } }`.
- **Discovery:** invoked via the task-runner recipe `just stack <subcommand>` OR direct `<task-runner-binary> harness <subcommand>` if the template prefers.
- **Promotion criterion:** boot→inspect→destroy cycle works on a clean checkout in ≤60s, without manual config.
- **Rust-first?** **Yes.** Single binary, cross-platform, no Node runtime.
- **Local pick:** [`ADR-0001-stack-manager`](../adrs/ADR-0001-stack-manager.md) — Rust binary using bollard + portpicker + shell-out to `docker compose`. Working Node/TS port from the lab landed under bead `n48.19` as the interim implementation while the Rust binary is built.

### 4.2 `inspector`

- **Purpose:** capture evidence — screenshots, recordings, keyframes, traces — that an agent or human can diff before/after a fix.
- **Required interface:** at minimum `screenshot-pair`, `record-flow`, `keyframe-grid`. Each MUST emit artifacts under `<project>/.harness/evidence/<run-id>/` and print the run-id to stdout.
- **Discovery:** invoked directly or via `just inspect <subcommand>`.
- **Promotion criterion:** an agent following the `before-after-evidence` skill can produce a reviewable bundle with no human steps.
- **Rust-first?** **Mixed.** Playwright is Node-rooted; Rust wrappers (fantoccini, headless_chrome) exist but maturity vs Playwright is a research-pending question per CLAUDE.md rule #0. Reasonable posture: keep inspector as a Node shell-out, treat "is Rust + headless_chrome viable as inspector backend?" as a running-experiments probe.
- **Local pick:** [`ADR-0002-inspector`](../adrs/ADR-0002-inspector.md) — Playwright + ffmpeg, Node. Sources at `harness/inspector/`.

### 4.3 `hooks`

- **Purpose:** run guardrails on commit and provide test-runner wrappers that emit token-efficient output.
- **Required interface:** a config file at a well-known path that the plugin's bin honors. MUST support skipping per-language hooks when their files aren't staged.
- **Discovery:** `just hooks install` to register, plus auto-fire on git events.
- **Promotion criterion:** clean repo passes `just hooks run --all` in <10s on a typical workstation.
- **Rust-first?** **Yes for our glue; underlying tool is research-pending.** The hook config + wrappers we ship can be Rust. The hook engine itself (lefthook is Go) is a separate research-backed pick.
- **Local pick:** [`ADR-0003-hooks`](../adrs/ADR-0003-hooks.md) — lefthook v2.1.6, single Go binary. Config at `lefthook.yml`. Six pre-push gates active: typecheck-affected, test-affected, scripts-coverage, perf-gate, doc-validator, ubs-diff.

### 4.4 `telemetry-sdk`

- **Purpose:** instrument application code with OTEL signals (logs, metrics, traces).
- **Required interface:** a per-language entrypoint that initializes an OTEL provider pointing at the `observability-stack`'s OTLP endpoint. Configuration via env vars (`OTEL_*`).
- **Polyglot rule:** a template MAY mount multiple `telemetry-sdk` plugins (one per language). Each MUST honor the same OTLP endpoint and resource-attribute conventions.
- **Discovery:** imported by the app code; documented at template README.
- **Promotion criterion:** a `hello-world` app under the plugin's language emits a span and a log line that the harness's stack queries can retrieve via the canonical `observability-queries` flow.
- **Rust-first?** **No — by nature per-language.** Node apps use opentelemetry-node, Rust apps use opentelemetry-rust, Python uses opentelemetry-python. No way around it.
- **Local pick:** [`ADR-0004-telemetry-sdk`](../adrs/ADR-0004-telemetry-sdk.md). Node-side uses `@opentelemetry/sdk-node` + auto-instrumentations; Rust uses `opentelemetry` + `opentelemetry_sdk` + `opentelemetry-otlp`; Python uses `opentelemetry-distro` + `opentelemetry-instrumentation`.

### 4.5 `observability-stack`

- **Purpose:** receive OTEL signals and store them queryably (logs/metrics/traces).
- **Required interface:** a compose file or equivalent that exposes OTLP-gRPC + OTLP-HTTP endpoints and provides query endpoints for logs, metrics, traces.
- **Polyglot rule:** language-agnostic; this is infrastructure.
- **Discovery:** booted by `stack-manager`; queryable via the `observability-queries` agent-plugin skill.
- **Promotion criterion:** cold-boot <30s on a typical workstation; survives a `destroy → boot` cycle without lost state in the run window.
- **Rust-first?** **N/A.** This slot is infrastructure (docker-compose'd services), not a tool we author.
- **Local pick:** [`ADR-0005-observability-stack`](../adrs/ADR-0005-observability-stack.md) — OTEL Collector contrib → VictoriaLogs + VictoriaMetrics + VictoriaTraces. Compose at `harness/observability/compose.harness.yml`.

### 4.6 `sensors`

- **Purpose:** compute architectural-governance signals (coupling, abstractness, instability, distance-from-main-sequence, etc.) at known scopes (file/module/folder).
- **Required interface:** a CLI subcommand per adapter that emits `Reading[]` JSON on stdout. Reference schema is the `harness/sensors` aggregator; per-adapter shape may vary, but the merged output keys on `source`.
- **Polyglot rule:** adapters per language; aggregator language-agnostic.
- **Discovery:** `just sensors report` (full report), `just sensors gate` (pre-push fitness gate).
- **Promotion criterion:** adapter's output is consumed by the aggregator and contributes to a policy decision (gate pass/fail).
- **Rust-first?** **Yes** in the lab; this template ships a Node-ESM aggregator + adapters as the working interim. ADR-0017 records the decision to preserve all adapters in the catalog.
- **Plug-and-play?** **Yes.** Optional plugin — removable without breaking other slots.
- **Local picks:** [`ADR-0006-sensors`](../adrs/ADR-0006-sensors.md) (v0.2 adapter catalog) + [`ADR-0017-sensors-v03-apss-canonical`](../adrs/ADR-0017-sensors-v03-apss-canonical.md) (v0.3 APSS canonical, sentrux preserved). Four adapters wired today: dep-cruiser, ts-morph abstractness, ts-morph complexity, APSS topology.

### 4.7 `agent-plugins`

- **Purpose:** bundle agent-facing assets — skills, slash commands, subagents, hooks, memory conventions — into a reusable unit. Higher-level than just skills.
- **Required interface:** a directory under `.claude/` containing any of: `skills/`, `commands/`, `agents/`, `hooks/`, `memory/`. A plugin MAY ship only one subdir; structure is convention, not contract enforcement.
- **Vendor primacy:** the canonical source is `.claude/` (and `CLAUDE.md` for the root agent context). Other-vendor dirs (`.codex/`, `.cursor/`) and files (`AGENTS.md`, `GEMINI.md`, etc.) are symlinks pointing AT the canonical pair. Templates SHOULD provide a `just agents link` recipe to create the symlinks idempotently.
- **Discovery:** Claude reads `.claude/` natively. Other agents see their conventional name via symlink.
- **Promotion criterion:** a plugin loaded in two different agent vendors triggers correctly via its skill description in both.
- **Rust-first?** **N/A.** Files on disk; not a tool we author.
- **Local pick:** [`ADR-0007-agent-plugins`](../adrs/ADR-0007-agent-plugins.md). Skills under `.claude/skills/`.

### 4.8 `task-runner`

- **Purpose:** be the discoverable single entrypoint for project tasks. `<runner>` with no args lists every recipe with a one-line description.
- **Required interface:** a manifest file at template root + a binary that reads it. MUST work identically on macOS, Linux. SHOULD work on Windows.
- **Constraint:** no `.sh` or `Makefile` as primary entrypoint. The runner MAY shell out to language-native scripts; but the entrypoint surface is single-runner.
- **Discovery:** `<runner>` with no args; `<runner> <recipe>` to run.
- **Promotion criterion:** new contributor goes from clone → running stack in ≤3 commands documented in the template README.
- **Rust-first?** **Already.** `just` is a Rust binary.
- **Local pick:** [`ADR-0008-task-runner`](../adrs/ADR-0008-task-runner.md) — `just` v1.51.0. Recipes at `justfile`.

### 4.9 `secret-scanner`

- **Purpose:** detect credentials / API keys / private keys / tokens before they're committed or pushed. Defense-in-depth against the most common credential-leak failure mode.
- **Required interface:** a CLI that takes a list of files (or a directory) and exits non-zero with a structured stderr block if any high-confidence secret is detected. MUST be invocable from `lefthook` pre-commit (`{staged_files}`). MUST support a project-local allowlist / baseline file for known-safe false positives.
- **Discovery:** invoked via `just secret-scan` (template recipe) or directly from the pre-commit hook.
- **Promotion criterion:** the canonical "live AWS key in a commit message" test fixture is detected within 1s on a typical workstation; the project's baseline file format is stable enough to commit and review.
- **Rust-first?** **No — pick-driven.** The slot is implementation-agnostic; the current plugin (Gitleaks) is a Go binary that meets the zero-runtime-dep bar. A future custom Rust adapter is plausible but not load-bearing today — the maintenance cost of recreating a 700+ pattern detector outweighs the dep saving.
- **Plug-and-play?** **Yes.** The hook is wired via `lefthook.yml` and can be removed without breaking other slots. The decision doc captures the pick + when-to-re-evaluate.
- **Local pick:** [`ADR-0009-secret-scanner`](../adrs/ADR-0009-secret-scanner.md) — Gitleaks v8.24.2 (MIT, 6.41 MB single binary).

### 4.10 `doc-validator`

- **Purpose:** catch broken internal cross-references in markdown docs (retros → experiments, journals → SHAs, decision docs → standard). Doc rot is a slow-burning harness-quality failure mode; catching it pre-push beats discovering it months later.
- **Required interface:** a CLI that takes a directory and exits non-zero with a structured report of broken relative-path links + anchor links. External URL validation is OUT OF SCOPE (different cost shape — rate-limit handling, offline-mode, etc.).
- **Discovery:** invoked via `just doc-check` (template recipe) or directly from pre-push.
- **Promotion criterion:** scanning the template's `docs/` + `experiments/` + `README.md` set completes in <2s; finds no false positives on the current repo state.
- **Rust-first?** **Yes.** Custom Rust binary at `harness/doc-validator/`. Rationale: scope is narrow (internal cross-refs only), maintenance is trivial, zero language-runtime dep, fast.
- **Plug-and-play?** **Yes.** Optional pre-push step; removable without breaking other slots.
- **Local pick:** [`ADR-0010-doc-validator`](../adrs/ADR-0010-doc-validator.md). Implementation at `harness/doc-validator/` (Rust crate, 638 LOC, populated under bead `n48.6`). Pre-push hook scans markdown + APSS ADR01 shape + manifest cross-references; currently reports clean across 337 markdown links.

### 4.11 `versioning`

- **Purpose:** enforce coherent version → changelog mapping. Catch "I bumped the version but forgot CHANGELOG.md" and "I made breaking changes but kept the patch version" pre-push. Defense-in-depth for release discipline.
- **Required interface:** a CLI that takes a project root + a mode (`whole-repo` | `per-package`) and exits non-zero if: (a) the current version (as read from canonical manifest file — `Cargo.toml` `[workspace.package].version`, `package.json` `"version"`, `pyproject.toml` `[project].version`, etc.) has no corresponding entry in the CHANGELOG.md, (b) the version scheme is violated (default: semver), (c) git tags don't match manifest versions where applicable. MUST support an opt-out file (`.versioning-ignore`) for legitimate exceptions.
- **Discovery:** invoked via `just version-check` (template recipe) or directly from pre-push.
- **Promotion criterion:** scanning the template's own repo completes in <1s; catches the canonical "version bumped, changelog missed" test fixture; supports both modes.
- **Rust-first?** **Yes for the orchestrator binary** (`harness/versioning/` Rust crate, ported from the lab under bead `n48.20`), but the slot is tool-agnostic — the *plugin* is whatever does the underlying changelog generation. Default plugin pick: cocogitto.
- **Plug-and-play?** **Yes.** Removable via `lefthook.yml` edit + `cog.toml` delete; doesn't break other slots.
- **Local pick:** [`ADR-0011-versioning`](../adrs/ADR-0011-versioning.md) — [cocogitto](https://github.com/cocogitto/cocogitto) v6+. Rust binary, conventional-commits toolbox with bump + changelog generation.

## 7. Versioning

- **Standard** follows semver. v0.x.y is unstable; breaking changes can happen on any bump until v1.0.0.
- **Plugins** independently versioned. A plugin MUST declare which Standard version it targets.
- **Templates** independently versioned. A template MUST declare which Standard version + which plugin versions it composes. This template currently pins `standard: "0.2"` in [`../../harness.manifest.json`](../../harness.manifest.json).

## 8. Open questions (this fork's working set)

The lab's broader v0.2 open-questions list lives in [`agentic-harness-lab/docs/journal/2026-05-14-polyglot-template-parked-decisions.md`](https://github.com/NeuralEmpowerment/agentic-harness-lab/blob/main/docs/journal/2026-05-14-polyglot-template-parked-decisions.md). The subset still relevant to this consumer fork:

- **Per-slot canonical binary naming** — does the `inspector` plugin need a stable binary name across templates? Today this template uses `harness/inspector/bin/inspector`; lab uses the same.
- **`agent-plugins` discovery** — auto-discover under `.claude/` (current behavior) or require explicit listing in a manifest? Tracked alongside bead `create-harness-app-n48.9` (install upstream `harness-engineering` Claude plugin).
- **`sensors` Reading schema** — belongs in the Standard or a separate sub-Standard? The APSS topology contract (ADR-0017) is one answer; the lab's deferred plugin-protocol at `harness/sensors/docs/plugin-protocol.md` is another. Re-probe when a second sensor plugin needs non-APSS opinions.
- **Windows symlink fallback** — copy? junction? template-level branch? Deferred to v0.3.

## 9. Non-goals (v0.1 — transports verbatim)

- Not specifying *which* package manager, linter, formatter, test runner, telemetry SDK, or hooks tool a template must use. Those are per-template, research-backed decisions.
- Not specifying CI integration. Templates MAY ship CI configs; the Standard doesn't mandate one.
- Not specifying release/deploy pipelines. Out of scope for v0.1.
- Not specifying license, security, or compliance posture. Those are template-level concerns.
